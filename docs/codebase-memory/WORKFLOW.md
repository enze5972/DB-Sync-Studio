# WORKFLOW

本文件定义后续所有任务的低 token 标准执行流程。

## 1. 每次任务的标准执行流程

1. 先读 `AGENTS.md`。
2. 再读 `docs/codebase-memory` 中与当前任务相关的上下文文件。
3. 优先读模块级 memory，再读任务相关 memory。
4. 用关键词搜索定位相关代码、配置、路由、表名、类名、方法名。
5. 只打开搜索结果中最相关的源码文件。
6. 先形成最小可行理解，再做最小变更。
7. 变更后只做必要验证。
8. 完成后更新相关 `docs/codebase-memory` 文件。

## 2. 文件读取预算

- 源码初始读取最多 5 个文件。
- 上下文优先读取 memory 文件，不计入源码预算。
- 不要为了理解全貌而扩大到整个目录。
- 不要重复读取无变化文件。
- 超出预算前必须先暂停说明。

## 3. 搜索优先策略

- 先搜索，再打开文件。
- 搜索关键词优先使用：
  - 类名
  - 方法名
  - API 路径
  - 表名
  - 配置 key
  - 页面路由
  - 错误信息
  - 日志关键字
- 只读命中的最相关文件，不做目录级扫读。

## 4. 输出长度限制

- 默认简短输出，除非用户明确要求详细解释。
- 不输出完整文件内容或大段 diff。
- 每次回复控制在 15 行以内。

## 5. memory 更新规则

- 每完成一个阶段后更新相关 `docs/codebase-memory` 文件。
- 优先更新：
  - `CONTEXT_COMPACT.md`
  - `IMPLEMENTATION_STATUS.md`
  - `NEXT_TASKS.md`
  - 必要时更新对应模块 memory
- 更新内容只写对后续开发有价值的信息。

## 6. 什么时候允许扩大扫描范围

- 当前任务相关的搜索结果不足以定位问题时。
- 需要确认某个行为的唯一实现链路时。
- 需要验证迁移、路由、入口、仓储或配置之间的直接关系时。
- 但必须先说明已经读了哪些文件，以及还要读哪些文件。

## 7. 什么时候必须先暂停并说明原因

- 源码读取达到 5 个仍无法继续判断时。
- 上下文不足且继续猜测会影响正确性时。
- 当前任务需要读无关模块才能推进时。
- 发现与已有约束冲突时。
- 需要用户确认但当前信息无法推断时。

## 8. 任务完成后的固定输出格式

每次任务完成后，只输出：

1. 当前做了什么
2. 修改了哪些文件
3. 测试或构建结果
4. 是否有风险
5. 下一步建议

## 9. 执行约束

- 不修改业务代码，除非任务明确要求。
- 不读取无关目录。
- 不扩大到全仓库扫描。
- 不长篇解释项目背景。
- 以最小正确变更为目标。
